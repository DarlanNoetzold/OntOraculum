# OntOraculum

OntOraculum is an ontology-based framework for monitoring, analyzing, and optimizing performance metrics in smart environments. The ontology models relationships across multiple domains, including hardware, software, network, energy, and security, enabling semantic reasoning, anomaly detection, and validation through SWRL rules and SPARQL queries.

This repository provides the ontology artifacts used in the study, including different ontology versions, expanded models with instances, and inferred versions generated after reasoning.

## Repository Purpose

The goal of this repository is to make the ontology artifacts used in the OntOraculum study publicly available and reproducible. In particular, this repository supports:

- inspection of ontology classes, properties, and individuals;
- execution of semantic reasoning using the Pellet reasoner in Protégé;
- inspection of SWRL rules embedded in the ontology;
- reproduction of validation workflows based on competency questions and semantic queries.

## Repository Contents

The repository currently contains multiple `.owl` files corresponding to different stages or versions of the ontology development process.

### Main ontology files

- `metrics_performance.owl`  
  Early or base ontology version containing the main semantic structure for performance metrics.

- `metrics_performance_2.owl`  
  Revised ontology version with additional refinements.

- `ultimate_performance_metrics_v5_classes.owl`  
  Ontology version focused on class definitions and structural modeling.

- `ultimate_performance_metrics_v5_expanded.owl`  
  Expanded ontology version including a broader semantic structure.

- `ultimate_performance_metrics_v5_expanded_instances.owl`  
  Expanded ontology with generated instances representing simulated monitoring scenarios.

- `ultimate_performance_metrics_v5_expanded_instances_inf.owl`  
  Inferred version of the ontology after reasoning.

- `ultimate_performance_metrics_v5_expanded_instances_inf_2.owl`  
  Updated inferred version of the ontology after reasoning and refinement.

- `ultimate_performance_metrics_v5_refined.owl`  
  Refined ontology version used during iterative development.

### Supporting files

- `*.properties`  
  Protégé-related auxiliary configuration files. These are not required to inspect the ontology structure, but they may be generated by the ontology editing environment.

- `instances_table.tex`  
  LaTeX table used to summarize ontology instances or ontology statistics in the manuscript.

## Recommended File for Reproduction

For users who want to inspect the ontology together with generated individuals and inferred results, the most practical starting points are:

- `ultimate_performance_metrics_v5_expanded_instances.owl` for the ontology with generated instances before inference;
- `ultimate_performance_metrics_v5_expanded_instances_inf_2.owl` for the ontology after reasoning and inferred classifications.

## Software Requirements

To reproduce the ontology workflow, we recommend the following environment:

- [Protégé 5.5.0](https://protege.stanford.edu/)
- Pellet reasoner available through Protégé
- SWRL support in Protégé (e.g., SWRLTab)
- Optional: SPARQL query support in Protégé, depending on the installed plugins

## How to Use OntOraculum in Protégé

### 1. Clone or download the repository

```bash
git clone https://github.com/DarlanNoetzold/OntOraculum.git
```

Or download the repository as a ZIP file from GitHub.

### 2. Open the ontology in Protégé

1. Launch Protégé 5.5.0.
2. Go to **File > Open**.
3. Select one of the ontology files from this repository.
4. For full reproduction of the validation scenario, start with:
   - `ultimate_performance_metrics_v5_expanded_instances.owl`, or
   - `ultimate_performance_metrics_v5_expanded_instances_inf_2.owl`.

### 3. Inspect the ontology structure

After opening the ontology, inspect the following Protégé tabs:

- **Classes**: view the ontology hierarchy;
- **Object Properties**: inspect semantic relationships among entities;
- **Data Properties**: inspect numerical and descriptive attributes such as metric values;
- **Individuals**: inspect generated instances representing monitoring scenarios.

### 4. Run semantic reasoning with Pellet

To reproduce logical inference:

1. In Protégé, select **Reasoner**.
2. Choose **Pellet** as the active reasoner.
3. Start the reasoner.
4. Wait for classification and inference completion.
5. Inspect inferred class memberships and relations in the **Classes** and **Individuals** views.

This reasoning step enables the ontology to classify scenarios such as high CPU usage, high latency, excessive power consumption, anomaly detection, and security-related conditions.

### 5. Inspect SWRL rules

The ontology includes SWRL rules used to infer critical conditions and relationships among monitored metrics.

To inspect them:

1. Open the **SWRLTab** in Protégé.
2. Review the available rules associated with performance, QoS, anomaly detection, energy, and security scenarios.
3. Compare inferred results before and after reasoner execution.

Examples of scenarios represented in the ontology include:

- high CPU usage;
- low memory availability;
- unbalanced network latency;
- high energy consumption;
- high error rates;
- potential intrusion attempts;
- malware detection;
- accurate detection scenarios.

### 6. Reproduce validation with queries

The study validates the ontology using competency questions and semantic queries.

To reproduce this process:

1. Open the ontology in Protégé.
2. Ensure individuals are loaded.
3. Run the reasoner.
4. Execute the SPARQL queries associated with the competency questions described in the manuscript.

The validation workflow investigates relationships such as:

- network latency and QoS;
- uptime and stability;
- CPU, memory, and throughput;
- energy efficiency and power consumption;
- intrusion attempts and stability;
- congestion and service degradation.

If your Protégé installation does not include SPARQL query support by default, use a compatible plugin or an RDF/SPARQL environment that can load the ontology file and execute the queries externally.

## Instance Generation and External Simulator

The monitoring instances used in the ontology were generated from a simulator for sensor data management scenarios. That simulator is available in a separate public repository:

- [SensorSimulator](https://github.com/DarlanNoetzold/SensorSimulator)

The simulator produces synthetic metric values representing realistic operational conditions in smart environments. These values are then mapped into ontology individuals and evaluated through semantic rules and reasoning.

## Reproducibility Workflow

A minimal workflow to reproduce the ontology behavior is:

1. Obtain the ontology from this repository;
2. Obtain the simulation source or generated scenario data from the simulator repository;
3. Open the ontology in Protégé;
4. Load the ontology version containing individuals;
5. Run the Pellet reasoner;
6. Inspect SWRL rules in SWRLTab;
7. Execute validation queries based on the competency questions;
8. Compare inferred classifications and query results with those reported in the manuscript.

## Notes on File Versions

This repository contains multiple ontology versions produced during the development process. File names reflect different stages, including:

- class-focused versions;
- expanded versions;
- versions with generated instances;
- inferred versions after reasoning.

For reproducibility purposes, users should prefer the ontology version with instances and, when needed, the inferred version.
